# Phase 01: Dual-Agent Document Review

## Capability
Run one document task through two agents and save both outputs for inspection.

## Version name
`dual-agent-document-review-v0.1-draft`

## Why this phase
- proves the orchestration loop
- keeps the scope narrow
- creates immediate value
- avoids risky auto-merge behavior

## In scope
- one source document
- one task payload
- one Codex response
- one Claude response
- one run folder
- one run metadata file

## Out of scope
- automatic merging
- in-place editing of the source file
- multi-document workflows
- dashboard implementation
- memory system

## Input contract
- `task_id`
- `project_id`
- `source_document_path`
- `goal`
- `constraints` (optional, defaults to an empty list)
- `output_root`

`run_id` is generated by the workflow at trigger time and is not supplied by the caller.

## Agent response contract
Each agent should return:
- `summary`
- `proposed_text`
- `change_notes`
- `open_questions`

## Stage and logging map
For Phase 01, vendor or model identity is recorded in `agent_outputs[].agent_role`, while the workflow position is recorded in `current_stage`.
During the parallel model-call portion of this phase, `current_stage` should be set to `agent_review`.

| Workflow moment | `current_stage` | `agent_role` |
|---|---|---|
| Payload intake and source load | `intake` | none |
| Parallel OpenAI and Claude review calls | `agent_review` | `codex` or `claude` |
| Human inspection of saved outputs | `approval` | none |
| Final run record closeout | `finalize` | none |

## Proof artifacts
- source snapshot
- Codex output file
- Claude output file
- run metadata file

## Acceptance test
This phase is proven when:
1. The workflow accepts one document task.
2. Both agents return output.
3. Both outputs contain a non-empty `proposed_text` field.
4. Both outputs conform to the response contract.
5. Both outputs are saved separately.
6. The original source document is unchanged.
7. A run record is saved with valid `started_at` and `ended_at` timestamps.
8. A duplicate `run_id` does not silently overwrite an existing run folder.

## Review questions
- Did both agents stay inside the same response contract?
- Was the source preserved?
- Is the run folder easy to inspect?
- Are we ready to add approval as Phase 02?

## Promotion
Promote to `dual-agent-document-review-v0.1-stable` only after a successful test run is saved and reviewed.
